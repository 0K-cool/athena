# Phase 4.5: Deep Analysis — 0-Day Hunting Capability

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Add autonomous 0-day discovery capability to ATHENA via two new specialized agents (DA + PX) operating in a tight hypothesis→probe→analyze loop between vulnerability scanning and exploitation.

**Architecture:** DA (Opus) generates hypotheses and analyzes responses while PX (Sonnet) executes rapid probes and fuzzing. They communicate bilaterally without ST bottleneck. Tiered escalation (confidence thresholds) gates when findings reach exploitation.

**Tech Stack:** Claude Agent SDK (Opus + Sonnet), Neo4j knowledge graph, Kali MCP tools, direct HTTP via `execute_command`

---

## Section 1: Agent Architecture

### DA — Deep Analysis Brain (Opus)

| Field | Value |
|-------|-------|
| **Model** | Opus |
| **Role** | Hypothesis generation, response analysis, payload crafting, pattern recognition |
| **Phase** | 4.5 (between Vulnerability Analysis and Exploitation) |
| **Why Opus** | Reasoning-heavy — must spot subtle anomalies, craft novel payloads, chain logic across multiple probe results |

DA does NOT execute probes. It thinks, hypothesizes, and directs PX.

### PX — Probe Executor (Sonnet)

| Field | Value |
|-------|-------|
| **Model** | Sonnet |
| **Role** | Fast HTTP loops, binary search oracles, fuzzing spray, Kali heavy scans |
| **Phase** | 4.5 |
| **Why Sonnet** | Execution-heavy — needs speed for rapid probe loops, not deep reasoning |

PX does NOT analyze results beyond basic pass/fail. It executes what DA directs and returns raw results.

### Communication

- **DA↔PX:** Bilateral messaging (direct, no ST bottleneck) for tight probe loops
- **DA→ST:** Escalation when confidence reaches thresholds (70%+ notify, 90%+ escalate)
- **ST→DA:** Phase gating — ST activates DA after WV/VF complete Phase 4
- **PX→Neo4j:** All probe results stored as `ProbeResult` nodes

### Dashboard Integration

## Section 2: DA Reasoning Model

### The Loop (max 5 iterations per target endpoint)

```
1. HYPOTHESIZE — Generate 3-5 hypotheses about what could be vulnerable
   Input: WV findings, service fingerprint, tech stack, CEI data
   Output: Ranked hypotheses with test plans

2. DESIGN PROBES — Craft specific payloads to test each hypothesis
   Output: Probe specifications for PX (URL, method, headers, body, expected response)

3. ANALYZE RESPONSES — Compare probe results against expected behavior
   Input: PX raw results (status codes, response bodies, timing, errors)
   Output: Updated confidence scores per hypothesis

4. REFINE — Narrow or pivot based on what was learned
   - Confirmed hypothesis → escalate
   - Partially confirmed → design deeper probes
   - Refuted → discard, generate new hypotheses

5. ESCALATE OR CONTINUE — Confidence-based decision
   - <50%: Dismiss hypothesis, try next
   - 50-70%: Keep probing (refine payloads, try variations)
   - 70-90%: Notify ST, continue probing for confirmation
   - 90%+: Escalate to ST for exploitation handoff + generate VDR
```

### Hypothesis Categories

DA generates hypotheses across these categories (not limited to):

- **Input validation gaps** — parameters that accept unexpected types/lengths
- **Logic flaws** — race conditions, TOCTOU, state machine bypasses
- **Authentication/authorization gaps** — privilege escalation, IDOR
- **Injection points** — SQL, command, template, LDAP, XPath
- **Information disclosure** — error messages, debug endpoints, stack traces

### Confidence Scoring

Each hypothesis gets a confidence score (0-100) updated after every probe round:

- **Response differential** — How different is the response from baseline?
- **Timing anomaly** — Does the response time change with payload complexity?
- **Error signature** — Does the error reveal internal state?
- **Behavioral consistency** — Does the anomaly reproduce reliably?

## Section 3: PX Execution Modes

PX operates in 4 modes, selected by DA based on what's being tested:

### Mode 1: Rapid Probe
- **What:** Single HTTP requests with crafted payloads
- **Tool:** `execute_command` → `curl` with precise headers/body
- **Speed:** 1 request at a time, immediate response to DA
- **Use case:** Initial hypothesis testing, response differential analysis

### Mode 2: Binary Search Oracle
- **What:** Bisect parameter space to find exact boundary
- **Tool:** `execute_command` → scripted curl loops
- **Speed:** O(log n) requests to find threshold
- **Use case:** Finding exact length limits, character restrictions, WAF bypass boundaries

### Mode 3: Fuzzing Spray
- **What:** Targeted fuzzing with wordlists/mutations
- **Tool:** `execute_command` → `ffuf`, custom scripts
- **Speed:** Batch execution, return summary to DA
- **Use case:** Parameter discovery, hidden endpoint enumeration, input mutation testing

### Mode 4: Kali Heavy
- **What:** Full Kali tool execution for deep scanning
- **Tool:** Kali MCP tools (sqlmap, nuclei custom templates, nmap scripts)
- **Speed:** Slower, more thorough
- **Use case:** Confirming suspected injection points, running exploit-specific tools

### Phase 2 Enhancement: Playwright Browser
- **What:** Full browser automation for client-side vulnerabilities
- **Tool:** Playwright MCP
- **Use case:** DOM XSS, client-side logic flaws, JavaScript-based vulnerabilities
- **Status:** Deferred to Phase 2 implementation

## Section 4: Attack Methodology

DA's hypothesis generation is guided by 5 attack methodology categories:

### Category 1: Input Boundary Probing
- Test every input parameter with boundary values
- Type confusion (string where int expected, array where string expected)
- Length boundaries (0, 1, max, max+1, massive)
- Encoding variants (URL, double-URL, Unicode, null bytes)
- Goal: Find inputs that cause unexpected behavior

### Category 2: Logic Flow Analysis
- Map the application's state machine
- Test transitions that shouldn't be possible (skip steps, repeat steps)
- Race condition probes (parallel requests to same endpoint)
- TOCTOU (time-of-check-to-time-of-use) patterns
- Goal: Find logic that can be subverted

### Category 3: Authentication & Authorization Bypass
- IDOR testing (increment/decrement IDs, swap user contexts)
- JWT manipulation (algorithm confusion, claim tampering)
- Session handling (fixation, prediction, concurrent sessions)
- Privilege escalation (horizontal + vertical)
- Goal: Access resources/actions beyond authorized scope

### Category 4: Injection Discovery
- SQL injection (error-based, blind boolean, time-based, UNION)
- Command injection (semicolons, pipes, backticks, $() substitution)
- Template injection (SSTI — Jinja2, Twig, Freemarker, Pebble)
- LDAP/XPath/Header injection
- WAF bypass techniques (encoding, chunked transfer, HTTP/2 smuggling)
- Goal: Execute attacker-controlled code/queries

### Category 5: Information Leakage Exploitation
- Error message analysis (stack traces, SQL errors, path disclosure)
- Debug endpoint discovery (/debug, /trace, /actuator, /swagger)
- Response header analysis (server versions, framework fingerprints)
- Timing side channels (username enumeration, conditional delays)
- Goal: Extract information that enables deeper attacks

## Section 5: Integration with Existing Architecture

### PTES Phase Flow (Updated)

```
Phase 1: Pre-engagement → Phase 2: Recon (AR) → Phase 3: Threat Modeling (ST)
→ Phase 4: Vulnerability Analysis (WV + VF) → Phase 4.5: Deep Analysis (DA + PX)
→ Phase 5: Exploitation (EX) → Phase 6: Post-Exploitation (VF) → Phase 7: Reporting (RP)
```

### Activation

- ST activates DA after Phase 4 completes (WV scan + VF verification done)
- DA receives: all WV findings, VF verification results, service fingerprints, CEI data
- DA autonomously activates PX as needed (no ST approval required for probing)
- Exploitation handoff: DA→ST→EX (ST gates the transition, HITL in client mode)

### Tiered Autonomy (Phase 4.5)

| Mode | DA Behavior | PX Behavior | Escalation |
|------|------------|------------|------------|
| **CTF/Lab** | Full autonomy — probe everything | Full autonomy — all tools | Auto-escalate to EX at 90%+ |
| **Client** | Full autonomy for analysis | HITL gate for destructive probes | ST + HITL approval at 90%+ |
| **Client Auto** | Full autonomy | Full autonomy | Auto-escalate to EX at 90%+ |

Note: Probing is non-destructive by default. Only destructive probes (SQLMap write tests, active exploitation) trigger HITL in client mode.

### Neo4j Knowledge Graph Extensions

New node types:
- `Hypothesis` — DA's hypotheses with confidence scores and status
- `ProbeResult` — PX's raw probe results linked to hypotheses
- `ZeroDayFinding` — Confirmed 0-days (90%+ confidence) with full VDR data

New relationships:
- `(Hypothesis)-[:TESTS]->(Service)` — what's being tested
- `(ProbeResult)-[:VALIDATES]->(Hypothesis)` — evidence chain
- `(ZeroDayFinding)-[:DISCOVERED_IN]->(Engagement)` — engagement linkage
- `(ZeroDayFinding)-[:AFFECTS]->(Service)` — what's vulnerable

### CEI (Cross-Engagement Intelligence) Integration

- **TechniqueRecords:** 0-day techniques recorded with target stack fingerprint
- **FalsePositiveRecords:** Failed hypotheses recorded to avoid repeating
- **Experience Briefs:** DA receives past 0-day patterns for similar tech stacks
- DA prioritizes hypothesis categories that have historically yielded findings on similar stacks

Dashboard agent grid expands from 6 to 8 agents:

```

```

Each with standard LED states: idle (gray), running (green pulse), completed (solid green), error (red).

## Section 6: Responsible Disclosure Documentation

When DA confirms a 0-day (confidence 90%+), the system auto-generates a **Vulnerability Disclosure Report (VDR)** with everything a vendor needs to replicate, validate, and fix.

### VDR Structure

```
ATHENA-VDR-{YYYY}-{NNN}/
├── summary.md              # Executive summary + CVSS 4.0 score
├── technical-details.md    # Full technical writeup
├── reproduction/
│   ├── steps.md            # Step-by-step reproduction guide
│   ├── payloads/           # Exact payloads used (sanitized for safe handling)
│   ├── requests.har        # HTTP archive of the exploit chain
│   └── environment.md      # Target stack, versions, OS, config
├── evidence/
│   ├── screenshots/        # Before/during/after screenshots
│   ├── responses/          # Raw HTTP responses proving impact
│   └── diff-analysis.md    # Normal vs exploited behavior comparison
├── impact-assessment.md    # What an attacker could achieve
├── remediation/
│   ├── recommended-fix.md  # Specific fix guidance (code-level if possible)
│   ├── workarounds.md      # Temporary mitigations
│   └── references.md       # Related CVEs, CWEs, OWASP mappings
└── timeline.md             # Discovery timestamp, disclosure timeline
```

### Technical Details Fields

| Field | Purpose |
|-------|---------|
| **Affected Component** | Exact endpoint, function, parameter |
| **Attack Vector** | Network/Local/Adjacent + authentication required |
| **Root Cause** | Why the vulnerability exists (missing validation, logic flaw, etc.) |
| **Exploit Chain** | Ordered steps from initial access to impact proof |
| **Payload Details** | Exact inputs that trigger the vulnerability |
| **Response Analysis** | How the vulnerable response differs from normal |
| **Stack Fingerprint** | Software versions, frameworks, middleware detected |

### Reproduction Guarantee

Every VDR includes a self-contained reproduction section that a vendor engineer can run in their lab environment to confirm the finding independently:

- Exact URLs, exact payloads, exact expected responses
- Environment requirements (tools needed, network access)
- Step-by-step with expected output at each step
- HAR file for HTTP-based vulnerabilities (importable into any proxy tool)

### Auto-Generation Flow

1. DA reaches 90%+ confidence → triggers VDR generation
2. PX replays the minimal exploit chain, capturing HAR + screenshots
3. DA writes technical analysis, root cause, and remediation guidance
4. RP (Reporting Agent) formats into the VDR structure
5. Stored in Neo4j as `ZeroDayFinding` node linked to the engagement
6. Operator reviews before any external disclosure

### Disclosure Timeline Tracking

- Discovery date, operator notification date recorded automatically
- Configurable disclosure window (default: 90 days, adjustable per engagement)
- ATHENA tracks the timeline and reminds the operator at 30/60/90 days
- `timeline.md` updated as disclosure progresses

## Non-Functional Requirements

- **Performance:** DA reasoning loop completes within 60s per iteration; PX probe execution within 30s per batch
- **Safety:** All probing is non-destructive by default; destructive probes require mode-appropriate approval
- **Storage:** All hypotheses, probes, and findings persisted in Neo4j for CEI
- **Observability:** DA/PX activity visible in real-time dashboard event feed
